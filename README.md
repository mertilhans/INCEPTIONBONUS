## Description

Inception is a 42 project about Docker and system administration. The goal is to build a real web infrastructure where multiple services run in their own containers and work together. Everything must be built from scratch using Debian — no ready-made Docker images allowed (except Debian itself).

I built the mandatory part (NGINX, WordPress, MariaDB) and all 5 bonus services (Redis, FTP, Static website, Adminer, Portainer). The whole infrastructure runs behind NGINX with HTTPS only, using TLSv1.2 and TLSv1.3.

This project taught me a lot. I had to understand how each service actually works instead of just pulling a pre-built image and hoping for the best.

---

## Services

### Mandatory

**NGINX** — The only entry point to the infrastructure. It handles SSL with a self-signed certificate generated by OpenSSL. Only port 443 is open. Everything else is hidden inside the Docker network.

**WordPress + PHP-FPM** — The main website. PHP-FPM handles PHP on port 9000 and NGINX talks to it with FastCGI. WordPress is installed automatically with WP-CLI on first boot. There are two users: one admin (username does not contain "admin") and one author.

**MariaDB** — The WordPress database. It only lives inside the Docker network. Nothing from outside can reach it directly.

### Bonus

**Redis** — Object cache for WordPress. Instead of querying the database on every request, WordPress keeps frequently used data in Redis memory. Much faster. Installed and enabled automatically with WP-CLI.

**FTP Server (vsftpd)** — Gives direct file access to the WordPress volume. Uses passive mode on ports 21100–21110.

**Static Website** — My personal portfolio. It was already built as a React app on GitHub Pages. Since the build output is pure HTML, CSS and JavaScript, I just clone the repository during the Docker build and serve it with NGINX. Available at `/portfolio/`. No PHP, exactly what the subject asked for.

**Adminer** — A single PHP file that gives a browser UI for the MariaDB database. Simple, lightweight, and useful for database inspection. Available at `/adminer/`.

**Portainer** — A web UI for managing Docker containers. I chose this as my "service of your choice" because it is actually useful — you can see logs, inspect volumes, and check container status without typing long terminal commands. Available at `/portainer/`.

**1. Add the domain to your hosts file:**
```bash
echo "127.0.0.1 merilhan.42.fr" | sudo tee -a /etc/hosts
```

**2. Create the secrets folder and password files:**
```bash
mkdir -p secrets
echo "your_db_password"       > secrets/db_password.txt
echo "your_root_password"     > secrets/db_root_password.txt
echo "your_wp_admin_password" > secrets/wp_admin_password.txt
echo "your_wp_user_password"  > secrets/wp_user_password.txt
```

## Project Description

### Virtual Machines vs Docker

Before this project I knew what Docker was but I never thought deeply about why it is different from a VM. A virtual machine runs a full operating system — its own kernel, its own memory management, everything. This means strong isolation but heavy resource usage. Starting a VM takes minutes.

Docker containers share the host kernel. They only contain the application and its libraries. Starting a container takes seconds and uses much less memory. For this project with 8 separate services, using VMs would need 8 full operating systems running at the same time. With Docker, they all share the same kernel and start fast.

The trade-off is isolation. If a serious kernel vulnerability appears, containers are more at risk than VMs. But for a learning project and most development environments, Docker is the right choice.

| | Virtual Machine | Docker Container |
|---|---|---|
| Boot time | Minutes | Seconds |
| Memory usage | Very heavy | Light |
| Isolation level | Full OS | Process level |
| Best use case | Maximum security | Fast, lightweight services |

### Secrets vs Environment Variables

When I started this project I was putting everything in the `.env` file — database names, usernames, passwords, all together. That is a problem. If `.env` accidentally gets committed to git, all your passwords are public. This has happened to real companies and caused serious security incidents.

The right approach is to separate them. Non-sensitive configuration goes in `.env`. Actual passwords go in Docker secrets — separate files on the host machine, mounted into the container at `/run/secrets/`. The script reads the password at runtime:

```sh
DB_PASSWORD=$(cat /run/secrets/db_password)
```

The `secrets/` folder and `.env` are both in `.gitignore` so they never touch the git repository.

| | Environment Variables (.env) | Docker Secrets |
|---|---|---|
| What to store | Names, config, domain | Passwords, API keys |
| Risk if in git | Low | Catastrophic |
| How to read | `$VARIABLE` | `cat /run/secrets/name` |
| In .gitignore | Yes | Yes |

### Docker Network vs Host Network

`network: host` means a container shares the host machine's network completely. It can reach everything and everything can reach it. Convenient, but zero isolation.

I used a custom bridge network called `dev_net`. All containers join this network and talk to each other using the service name as a hostname — WordPress connects to MariaDB by calling `mariadb:3306`. From outside the Docker network, only NGINX is reachable on port 443. MariaDB, Redis, PHP-FPM — none of them are exposed.

The subject also forbids `network: host` directly, so this was not optional anyway.

| | Host Network | Custom Bridge Network |
|---|---|---|
| Container isolation | None | Yes |
| Service discovery | By IP only | By service name |
| External exposure | Everything | Only published ports |
| Allowed by subject | ❌ | ✅ |

### Docker Volumes vs Bind Mounts

A bind mount directly links a host path into a container like this:
```yaml
volumes:
  - /home/merilhan/data/wordpress:/var/www/wordpress
```

The subject says bind mounts are not allowed. But it also says the data must be stored inside `/home/merilhan/data/` on the host. This sounds like a contradiction.

The solution is named volumes with local driver options. Named volumes are managed by Docker — they appear in `docker volume ls`, they can be inspected, and Docker controls their lifecycle. With the right driver options, they can store data at a specific host path. This satisfies both requirements.

```yaml
volumes:
  wp_vol:
    driver: local
    driver_opts:
      type: none
      device: /home/merilhan/data/wordpress
      o: bind
```

| | Direct Bind Mount | Named Volume |
|---|---|---|
| Docker manages it | No | Yes |
| `docker volume ls` | Not shown | Shown |
| Portable | No | Yes |
| Allowed by subject | ❌ | ✅ |

## Challenges I Faced

**302 redirect loop** — WordPress was redirecting every request in a loop. Two problems at once: the install URL was `merilhan.42.fr` without `https://`, so WordPress stored `http://` in the database while NGINX only serves HTTPS. And I was using `sleep 20` to wait for MariaDB, which sometimes was not enough time. I replaced the sleep with a retry loop and fixed the URL to include `https://`.

**SSL certificate error in curl** — When building the WordPress image, curl could not download WP-CLI because `ca-certificates` was not installed yet. One line fix: add `ca-certificates` to the apt install command.

**FTP write error** — The adminer container tried to download its PHP file to a directory that did not exist yet. Fixed by adding `mkdir -p` before the curl command.

**Portainer timeout** — Portainer has a 5-minute window to set up an admin account on first start. I learned this the hard way when I got a timeout page every time.

- **Docker secrets** — I was not sure if the PDF meant real Docker Swarm secrets or docker-compose file secrets. Claude explained the difference clearly and helped me update all the shell scripts to read from `/run/secrets/` instead of environment variables.

- **nginx sub-path proxying** — Getting Portainer and Adminer to work at `/portainer/` and `/adminer/` through the main NGINX required understanding how `proxy_pass` handles URL rewriting. Claude helped me configure this correctly, including WebSocket support for Portainer.

- **Build errors** — The curl SSL error and the FTP directory error were fixed quickly with Claude's help identifying the root cause.
